Indicator & score philosophy — read before treating any of this as a grade

The four indicators above separate two different questions this skill used to blend into one number: whether the docs are accurate (Documentation fidelity) is a different axis from whether the architecture is sound (Architecture risk) — a repo can improve one and regress the other. Audit coverage says how much of the mandatory checklist actually got run (not how many problems were found); Evidence confidence says how much of what's reported rests on direct citation versus inference. A dash (—) means that indicator legitimately wasn't computed for this run (wrong mode, or an older report predating that field) — never a fabricated zero.

Debt index (secondary, 0-100) is the original heuristic, debt-weighted signal, kept for continuity with earlier reports. Only risk, misaligned, and gap findings count against it, weighted by severity (high=6, medium=3, low=1, info=0) and summed per dimension, each dimension capped at 15, before those capped totals are added together; strength findings give a small capped bonus (+1 each, max +5). Deeper review surfaces more findings, so it isn't comparable across repos or across runs at different scope/depth — use it to track one repo's trend, not to rank systems against each other. Findings aren't independent (several may share one root cause and still get penalized separately). It reflects severity as judged during this review, not business impact or likelihood.

Risk
Data Architecture · High
Task's JPA @Id (a single column, id) does not match its real DDL primary key (composite: CreatedDate, Id) -- and the entity has no field for CreatedDate at all
This is more severe than a simple 'single column vs composite key' mismatch: the entity doesn't merely fail to include CreatedDate in its @Id, it has no mapped property for CreatedDate at all, even though CreatedDate (an int, distinct from the CreatedAt datetime column) is half of the real primary key and the partition key. Hibernate's identity/equality/first-level-cache model is keyed on `id` alone; two Task rows with the same id but different CreatedDate (e.g. an id reused across partitions, or a future recurring-task design that revisits an id) collide as 'the same entity' from the ORM's point of view, and the entity can't even express which partition a given persistent instance belongs to.
Recommendation: Map CreatedDate as an explicit entity field and move to a composite key (@IdClass or @EmbeddedId) that matches the real DDL primary key, or document/enforce (e.g. via a repository-layer invariant) why id alone is guaranteed globally unique across all partitions despite not being the actual database key.

Risk
Scalability · High
task-events consumer concurrency (=1) and TaskCleanupJob's worker-count (=1) are hardcoded literals with no environment override, on paths the vision doc gives concrete growth targets for
The vision doc states a concrete, calibratable target: 50 msg/s sustained on task-events today, growing 3x within 12 months (~150 msg/s) as the reminder feature exits beta. task-events runs on a hardcoded single-threaded consumer (concurrency=1) with no override -- every message is processed serially on one thread today, and that ceiling doesn't move without a code change and redeploy, exactly as growth is expected to triple the load. TaskCleanupJob's worker-count=1 has the identical pattern. Both sit next to notification.batch-size in the same file, which someone clearly already made env-overridable -- so the inconsistency isn't a blanket 'this codebase never externalizes config,' it's specifically these two capacity numbers that were left hardcoded.
Recommendation: Externalize task-events.concurrency and task.cleanup.worker-count the same way notification.batch-size already is, and load-test the consumer at the stated 150 msg/s (3x) target before the reminder feature's beta-to-GA rollout to confirm single-thread processing time per message actually clears that bar.

The vision doc sets a p99 < 300ms budget for synchronous REST endpoints, but bulk-import has no request-size guardrail at all -- a caller can submit an arbitrarily large task list in one synchronous call, which both risks blowing the latency budget for that request and, since bulk-import is a REST thread not a scheduled worker, has no notion of 'batch size' to tune the way notification.batch-size does for the messaging side.

Risk
Extensibility · Medium
Bulk-creating tasks (bulkImport) and creating a single task (create) are two structurally different code paths for the same underlying concern -- 'persist a task' -- one going through TaskService.create(), the other bypassing it entirely
Today both paths are equivalent because create() is a pure pass-through with no logic of its own -- but create() is exactly the kind of method that's meant to be the single place per-task business logic (validation, defaulting, event emission) lands. Because bulkImport() calls taskRepository.save() directly instead of delegating to create() per item, any future logic added to create() silently won't apply to bulk-imported tasks unless someone remembers to add it in both places. This is the same shape of inconsistency the pattern-doc-adoption check looks for, just in a place with no pattern doc to catch it.
Recommendation: Have bulkImport() call create() per item (or extract the per-task logic both paths need into one shared method) so the two paths can't drift the moment either one grows real logic.
