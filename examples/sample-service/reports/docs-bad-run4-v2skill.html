Indicator & score philosophy — read before treating any of this as a grade

The four indicators above separate two different questions this skill used to blend into one number: whether the docs are accurate (Documentation fidelity) is a different axis from whether the architecture is sound (Architecture risk) — a repo can improve one and regress the other. Audit coverage says how much of the mandatory checklist actually got run (not how many problems were found); Evidence confidence says how much of what's reported rests on direct citation versus inference. A dash (—) means that indicator legitimately wasn't computed for this run (wrong mode, or an older report predating that field) — never a fabricated zero.

Debt index (secondary, 0-100) is the original heuristic, debt-weighted signal, kept for continuity with earlier reports. Only risk, misaligned, and gap findings count against it, weighted by severity (high=6, medium=3, low=1, info=0) and summed per dimension, each dimension capped at 15, before those capped totals are added together; strength findings give a small capped bonus (+1 each, max +5). Deeper review surfaces more findings, so it isn't comparable across repos or across runs at different scope/depth — use it to track one repo's trend, not to rank systems against each other. Findings aren't independent (several may share one root cause and still get penalized separately). It reflects severity as judged during this review, not business impact or likelihood.

Risk
Data Architecture · High
Task's JPA `@Id` (single `id` column) does not match the real DDL primary key (composite `CreatedDate, Id`)
This is an architectural risk independent of what any doc claims (f1 covers the doc-accuracy angle separately): Hibernate's identity/equality/first-level-cache model is keyed on `id` alone. Two Task rows with the same id but different CreatedDate — plausible if an id were ever reused or if CreatedDate reflects a re-partitioning/reprocessing event — would collide as 'the same entity' from the ORM's point of view, even though the database itself would treat them as distinct rows. The risk is compounded by CreatedDate also being the partition key, so an entity-identity bug here interacts with partition placement.
Recommendation: Map the identity as a composite key (@IdClass or @EmbeddedId over id+createdDate) so ORM identity matches the real relational key, or confirm and document that id is guaranteed globally unique on its own (in which case the composite PK is defensive/partition-driven and the mismatch is intentional and should be documented as such, not merely accidental).

Risk
Observability · High
The failure-strategy=ignore drop path on both message consumers has zero output of any kind — not even a log line, let alone a metric — for a dropped message
Both consumers deliberately drop-and-move-on on failure (failure-strategy=ignore) rather than dead-lettering. Tracing that drop path to its output, as this check requires, finds nothing at all — no dead-letter producer, no reconciliation/retry sweep, no counter, and not even a log line noting a message was dropped. This is a sharper finding than 'no DLQ exists': a system that deliberately accepts data loss on this path needs at least a count of what it lost, and currently has no way for an operator to even know a drop happened, let alone how many.
Recommendation: At minimum, log a structured warning with the channel name and a correlation id on every failure-strategy=ignore drop; ideally add a counter per channel (task-events-dropped-total, task-reminders-dropped-total) so drop volume is visible on a dashboard, not just discoverable by reading connector config.
